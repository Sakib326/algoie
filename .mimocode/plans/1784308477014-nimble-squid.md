# Product Management MVP - Implementation Plan

## Context

The Algoie eCommerce platform has a basic product module with simple Product, Variant, Category, Brand, and Collection resources. The current implementation is missing many requirements: SEO fields, product types, tags, proper media management, inventory tracking, search/filtering, and a multi-step creation wizard. This plan enhances the existing module to a robust MVP that can scale toward a Shopify-like system.

## Gap Analysis: What Exists vs What's Needed

| Feature | Current State | Gap |
|---------|--------------|-----|
| Product basic info | name, description, status | Missing: slug, product_type, featured, new |
| SEO | none | Missing: slug, meta_title, meta_description |
| Variants | sku, price, compare_at_price, stock, option_values | Missing: cost_price, barcode, reserved_quantity, low_stock_threshold, position |
| Product type | none | Missing: simple vs variable |
| Tags | none | Missing entirely |
| Product images | string array on Product | No ordering, no alt text, no variant-specific images |
| Variant images | none | Missing entirely |
| Search/filtering | basic list, no filtering | Missing entirely |
| Creation wizard | single flat form | Needs multi-step wizard |
| Variant generation | manual | Needs auto-generation from attributes |

## 1. Domain Model (Enhanced)

### Existing Resources (Modified)

**Product** - Add: `slug`, `meta_title`, `meta_description`, `product_type` (:simple | :variable), `featured?`, `new?`, `attribute_definitions` (jsonb). Remove: `images` (replaced by ProductImage).

**Variant** - Add: `cost_price`, `barcode`, `reserved_quantity`, `low_stock_threshold`, `position`.

### New Resources

**Tag** - Reusable tags (e.g., "Summer", "Sale", "Organic"). Per-store, unique slug.

**ProductTag** - Join table linking products to tags.

**ProductImage** - Proper image resource with: `media_asset_id` (FK), `variant_id` (nullable FK), `alt_text`, `position`. Replaces the string array on Product.

## 2. Database Schema

### ALTER `products` table (tenant migration)

```sql
ALTER TABLE products
  ADD COLUMN slug TEXT,
  ADD COLUMN meta_title TEXT,
  ADD COLUMN meta_description TEXT,
  ADD COLUMN product_type TEXT NOT NULL DEFAULT 'simple'
    CHECK (product_type IN ('simple', 'variable')),
  ADD COLUMN featured BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN is_new BOOLEAN NOT NULL DEFAULT false,
  ADD COLUMN attribute_definitions JSONB DEFAULT '{}',
  DROP COLUMN images;
```

### ALTER `variants` table (tenant migration)

```sql
ALTER TABLE variants
  ADD COLUMN cost_price DECIMAL(18,2),
  ADD COLUMN barcode TEXT,
  ADD COLUMN reserved_quantity BIGINT NOT NULL DEFAULT 0,
  ADD COLUMN low_stock_threshold BIGINT NOT NULL DEFAULT 10,
  ADD COLUMN position BIGINT NOT NULL DEFAULT 0;
```

### CREATE `tags` table (tenant migration)

```sql
CREATE TABLE tags (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  slug TEXT NOT NULL,
  store_id UUID NOT NULL REFERENCES stores(id),
  inserted_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX tags_unique_slug ON tags (store_id, slug);
```

### CREATE `product_tags` table (tenant migration)

```sql
CREATE TABLE product_tags (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID NOT NULL REFERENCES products(id),
  tag_id UUID NOT NULL REFERENCES tags(id),
  inserted_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX product_tags_unique ON product_tags (product_id, tag_id);
```

### CREATE `product_images` table (tenant migration)

```sql
CREATE TABLE product_images (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id UUID NOT NULL REFERENCES products(id),
  variant_id UUID REFERENCES variants(id),
  media_asset_id UUID NOT NULL REFERENCES media_assets(id),
  alt_text TEXT,
  position BIGINT NOT NULL DEFAULT 0,
  inserted_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
CREATE INDEX product_images_product_idx ON product_images (product_id);
CREATE INDEX product_images_variant_idx ON product_images (variant_id);
```

## 4. Product Creation Workflow

### Current: Single flat form
### New: 7-step wizard

```
Step 1: Basic Info        → name, slug (auto-generated from name), description, product_type
Step 2: Category & Brand  → category_id, brand_id, featured?, new?
Step 3: Attributes        → attribute_definitions (e.g., Color: [Red, Blue])
                           → "Generate Variants" button (creates cartesian combinations)
Step 4: Pricing           → default price, cost_price, compare_at_price (applied to all variants)
Step 5: Inventory         → stock, low_stock_threshold, track_inventory?
Step 6: Images            → ProductImage management (reuses MediaManagerComponent)
Step 7: SEO               → slug, meta_title, meta_description, tags
Step 8: Review & Publish  → summary of all data, publish as draft/active

**State management:** Each step stores data in socket assigns (`@wizard_data`).
The wizard creates the product on the final step. Variant generation happens at Step 3.
```

### Wizard Implementation

- Separate LiveView: `ProductLive.Wizard` with a `step` assign (1-8)
- Each step validates its fields before allowing progression
- Product is created only on final "Publish" click
- Tags are created-or-found during the wizard
- Images are attached after product creation

## 5. Variant Generation Flow

### Input
User defines attribute groups in Step 3:
```json
{
  "Color": ["Red", "Blue", "Green"],
  "Size": ["S", "M", "L"]
}
```

### Processing (in `Algoie.Products.VariantGenerator`)
1. Compute cartesian product: `["Red", "Blue", "Green"] × ["S", "M", "L"]`
2. Generate 6 combinations: `{"Color": "Red", "Size": "S"}`, etc.
3. Auto-generate SKU for each: `{product-slug}-red-s` (lowercase, hyphenated)
4. Set position (0, 1, 2, ...) for ordering
5. Apply default price/cost to all variants
6. Set `track_inventory?: true`, `low_stock_threshold: 10`

### Output
A list of variant attrs ready for bulk creation via `Ash.create/3`.

## 6. Validation Rules

### Product
- `name`: required, 1-255 chars
- `slug`: required, unique per store, regex `^[a-z0-9]+(?:-[a-z0-9]+)*$`
- `product_type`: required, one of `[:simple, :variable]`
- `status`: required, one of `[:draft, :active, :archived]`, default `:draft`
- `meta_title`: max 60 chars (SEO best practice)
- `meta_description`: max 160 chars (SEO best practice)
- `attribute_definitions`: if product_type is `:simple`, must be empty or absent; if `:variable`, must have at least one attribute with at least one value

### Variant
- `sku`: required, unique per store
- `price`: required, positive decimal
- `compare_at_price`: if present, must be > price
- `cost_price`: if present, must be >= 0
- `stock`: must be >= 0
- `reserved_quantity`: must be >= 0, must be <= stock
- `low_stock_threshold`: must be >= 0
- `barcode`: optional, string

### Tag
- `name`: required, 1-100 chars
- `slug`: required, unique per store

### ProductImage
- `media_asset_id`: required
- `position`: required, integer >= 0
- `product_id`: required

## 11. Key Design Decisions

1. **Tags as a separate resource** - Not an array field. Tags need to be filterable, reusable, and potentially have their own metadata later. MVP uses a simple name+slug.

2. **ProductImage as a proper resource** - Replaces the string array. Supports ordering, alt text, and variant association. Links to MediaAsset (existing media library) via FK.

3. **Stock status computed, not stored** - `in_stock?` is computed from `stock > 0` and `track_inventory?`. `low_stock?` from `stock <= low_stock_threshold`. No stored status to avoid sync issues.

4. **Product type on Product** - `:simple` products get one auto-generated default variant. `:variable` products have multiple variants. This preserves the "every sellable item is a variant" invariant.

5. **Attribute definitions as JSONB** - Simpler than a separate table for MVP. Attribute definitions are always loaded with the product and rarely queried independently.

6. **Slug auto-generated from name** - Using `Slug.slugify(name)` with unique suffix if needed. Store-scoped unique index.

7. **Wizard state in socket assigns** - No database persistence until final step. Clean UX, no orphaned drafts.

8. **Variant generation as a pure function** - `VariantGenerator.generate/2` takes attribute definitions and base attrs, returns list of variant attrs. Easy to test, no side effects.
