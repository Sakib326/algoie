# Day 2 Implementation Plan — Commerce Core + Dashboard CRUD

## Overview

Build the commerce data model on top of the Day 1 foundation: Products, Categories, Brands, Collections, Variants, Customers, Coupons, Orders, OrderLineItems, an Order workflow module with concurrency-safe stock decrement, and LiveView dashboard CRUD for Products, Categories, Brands, and Orders.

---

## File Structure

All new files follow existing conventions (Ash resources, `domain:` attribute, `data_layer: AshPostgres.DataLayer`, `authorizers: [Ash.Policy.Authorizer]`, multitenancy `strategy(:context)`).

### Step 2: Product Resources

**`lib/algoie/products/product.ex`** — `Algoie.Products.Product`
- Fields: `name` (string, allow_nil?), `description` (string, nullable), `store_id` (uuid), `brand_id` (uuid, nullable), `category_id` (uuid, nullable), `status` (`:draft | :active | :archived`, default `:draft`), `images` (`:array, :string`, default `[]`), timestamps
- Multitenancy: `strategy(:context)`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `belongs_to :brand, Algoie.Products.Brand`, `belongs_to :category, Algoie.Products.Category`, `has_many :variants, Algoie.Products.Variant`, `has_many :collection_products, Algoie.Products.CollectionProduct`, `has_many :collections, through: :collection_products`
- Actions: `defaults([:read])`, `create :create` (accept name, description, store_id, brand_id, category_id, status, images), `update :update` (accept same), `destroy :destroy`
- Policies: same pattern as Store — create: `:system`, read/update: `ActorIsSystem | ActorHasAnyStoreAccess | ActorHasStoreAccess:staff`, destroy: `ActorIsSystem | ActorHasStoreAccess:owner`

**`lib/algoie/products/variant.ex`** — `Algoie.Products.Variant`
- Fields: `product_id` (uuid), `store_id` (uuid), `sku` (string), `price` (`:decimal, precision: 18, scale: 2`), `compare_at_price` (`:decimal, precision: 18, scale: 2, nullable`), `stock` (`:integer, default: 0`), `track_inventory?` (`:boolean, default: true`), `option_values` (`:map, default: %{}`), timestamps
- Multitenancy: `strategy(:context)`
- Identity: `unique SKU` within tenant — `identity(:unique_sku, [:store_id, :sku])`
- Relationships: `belongs_to :product, Algoie.Products.Product`, `belongs_to :store, Algoie.Stores.Store`, `has_many :order_line_items, Algoie.Orders.OrderLineItem`
- Validations: `validate_compare_at_greater_than_price` (compare_at_price > price, if both set), `validate_number(:stock, greater_than_or_equal_to: 0)`
- Actions: same pattern as Product
- Policies: same pattern as Product

**`lib/algoie/products/category.ex`** — `Algoie.Products.Category`
- Fields: `name` (string, allow_nil?), `store_id` (uuid), `parent_id` (uuid, nullable self-reference)
- Multitenancy: `strategy(:context)`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `belongs_to :parent, Algoie.Products.Category` (optional), `has_many :children, Algoie.Products.Category, foreign_key: :parent_id`, `has_many :products, Algoie.Products.Product`
- **Cycle prevention**: custom validation `validate_no_parent_cycle` — walk up parent chain from proposed parent_id, reject if current record appears
- Actions: same pattern

**`lib/algoie/products/brand.ex`** — `Algoie.Products.Brand`
- Fields: `name` (string, allow_nil?), `store_id` (uuid), timestamps
- Multitenancy: `strategy(:context)`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `has_many :products, Algoie.Products.Product`
- Actions: same pattern

**`lib/algoie/products/collection.ex`** — `Algoie.Products.Collection`
- Fields: `name` (string, allow_nil?), `description` (string, nullable), `store_id` (uuid), timestamps
- Multitenancy: `strategy(:context)`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `has_many :collection_products, Algoie.Products.CollectionProduct`, `has_many :products, through: :collection_products`
- Actions: same pattern

**`lib/algoie/products/collection_product.ex`** — `Algoie.Products.CollectionProduct`
- Fields: `collection_id` (uuid), `product_id` (uuid), timestamps
- Multitenancy: `strategy(:context)`
- Identity: `unique collection_product, [:collection_id, :product_id]`
- Relationships: `belongs_to :collection`, `belongs_to :product`
- Policies: `always()` (internal join table, like StoreStaff)

### Step 3: Customer & Coupon Resources

**`lib/algoie/customers/customer.ex`** — `Algoie.Customers.Customer`
- Fields: `name` (string), `email` (string, allow_nil?), `phone` (string, nullable), `store_id` (uuid), timestamps
- Multitenancy: `strategy(:context)`
- Identity: `unique_customer_email, [:store_id, :email]`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `has_many :orders, Algoie.Orders.Order`
- Actions: same pattern
- Policies: same as Product

**`lib/algoie/customers/coupon.ex`** — `Algoie.Customers.Coupon`
- Fields: `code` (string), `discount_type` (`:atom, :percent | :fixed`), `discount_value` (`:decimal, precision: 18, scale: 2`), `min_order_value` (`:decimal, precision: 18, scale: 2, nullable`), `starts_at` (`:utc_datetime, nullable`), `expires_at` (`:utc_datetime, nullable`), `usage_limit` (`:integer, nullable`), `times_used` (`:integer, default: 0`), `store_id` (uuid), timestamps
- Multitenancy: `strategy(:context)`
- Identity: `unique_coupon_code, [:store_id, :code]`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `has_many :orders, Algoie.Orders.Order`
- Validations:
  - `validate_number(:discount_value, greater_than: 0)`
  - Custom: if `discount_type == :percent`, validate `discount_value <= 100`
- Actions: same pattern (but note: `times_used` not in create/update accept lists — managed by workflow)
- Policies: same as Product
- **`valid_for_use?/1`** function: checks `starts_at <= now`, `expires_at > now` (exclusive), `usage_limit` nil or `times_used < usage_limit`

### Step 4: Order Resources

**`lib/algoie/orders/order.ex`** — `Algoie.Orders.Order`
- Fields: `store_id` (uuid), `customer_id` (uuid), `status` (`:atom, pending | pre_order | confirmed | fulfilled | cancelled`, default `:pending`), `total_amount` (`:decimal, precision: 18, scale: 2, default: Decimal.new(0)`), `coupon_id` (uuid, nullable), timestamps
- Multitenancy: `strategy(:context)`
- Relationships: `belongs_to :store, Algoie.Stores.Store`, `belongs_to :customer, Algoie.Customers.Customer`, `belongs_to :coupon, Algoie.Customers.Coupon`, `has_many :line_items, Algoie.Orders.OrderLineItem, foreign_key: :order_id`
- **No destroy action** — only read, create (internal), update (for status)
- **State machine validation**: custom change `validate_status_transition` that checks `{current_status, new_status}` against allowed transitions:
  ```
  pending → [:pre_order, :confirmed, :cancelled]
  pre_order → [:confirmed, :cancelled]
  confirmed → [:fulfilled, :cancelled]
  fulfilled → [] (terminal)
  cancelled → [] (terminal)
  ```
- Actions: `read :read` (primary), `create :create` (internal, accept store_id, customer_id, status, total_amount, coupon_id — not called directly for orders), `update :update_status` (accept status only, with state machine validation), no destroy
- Policies: same pattern as Product — create: `:system`, read/update: `ActorIsSystem | ActorHasAnyStoreAccess | ActorHasStoreAccess:staff`. No destroy action, no destroy policy. Status transitions require `:staff` level (staff handle day-to-day fulfillment).

**`lib/algoie/orders/order_line_item.ex`** — `Algoie.Orders.OrderLineItem`
- Fields: `order_id` (uuid), `variant_id` (uuid), `quantity` (`:integer`), `unit_price` (`:decimal, precision: 18, scale: 2`), timestamps
- Multitenancy: `strategy(:context)`
- Relationships: `belongs_to :order, Algoie.Orders.Order`, `belongs_to :variant, Algoie.Products.Variant`
- Validations: `validate_number(:quantity, greater_than: 0)`
- Actions: `defaults([:read])`, `create :create` (accept order_id, variant_id, quantity, unit_price — internal only)
- Policies: `always()` (internal, created by workflow)

### Step 5: Order Workflow Module

**`lib/algoie/orders/order_workflow.ex`** — `Algoie.Orders.OrderWorkflow`

```elixir
def create_order(tenant, attrs, actor) do
  # attrs: store_id, customer_id, variant_quantities: [{variant_id, quantity}], coupon_code (optional)
  
  # 1. Validate products are :active
  # 2. Validate stock for each variant
  # 3. Calculate totals
  # 4. Apply coupon if provided
  # 5. Reserve/decrease stock (FOR UPDATE row lock)
  # 6. Create Order + OrderLineItems
  # 7. Update coupon times_used if applicable
  
  # All inside Repo.transaction with row-level locks
end
```

Key implementation details:
- Use `Repo.transaction` wrapping the entire flow
- For stock decrement: `SELECT ... FOR UPDATE` on the Variant row before decrementing
- For coupon usage: `SELECT ... FOR UPDATE` on the Coupon row before incrementing `times_used`
- Use Ecto queries with `lock("FOR UPDATE")` for the locking pattern
- Reject `:draft`/`:archived` products
- Validate `times_used < usage_limit` inside the same locked transaction
- Snapshot the variant's current price into `unit_price`

### Step 7: Verification Script

The verification script (part of `seed_day2.exs`) runs all 18 scenarios:

1. Variant stock independence between siblings
2. Cross-store denial
3. SKU uniqueness within store vs across stores
4. Coupon math ($100 cart + 10% = $90)
5. Expired coupon rejected
6. Not-yet-started coupon rejected
7. Boundary: starts_at = now accepted, expires_at = now rejected
8. Stock validation (ordering > available → rejected)
9. Cross-store variant rejection in Order workflow
10. Price snapshot unchanged after variant price change
11. No destroy on Order (action must not exist); cancel_order works, staff-gated (any user with StoreStaff membership can cancel)
12. State machine: fulfilled→pending rejected, pending→confirmed succeeds
13. Category cycle prevention
14. Archived product ordering rejected
15. Full LiveView CRUD flow
16. SQL injection attempt → treated as literal string
17. Unauthorized write (staff of Store 2 → Store 1 product) → denied
18. No tenant context → fails closed

#### Dashboard Detail

**ProductLive.Index:**
- Table columns: Name, SKU (first variant), Price (first variant), Stock (sum of variants), Status (badge: draft/active/archived), Actions (Edit, Delete)
- Form fields: name, description (textarea), brand (select), category (select), status (select), images (text input, comma-separated URLs for now)
- Empty state: "No products yet. Create your first product."
- Delete: confirmation modal before destroy

**CategoryLive.Index:**
- Table columns: Name, Parent (name or "—"), Products (count), Actions (Edit, Delete)
- Form fields: name, parent_id (select from existing categories, excluding self and descendants for cycle prevention)
- Empty state: "No categories yet."

**BrandLive.Index:**
- Table columns: Name, Products (count), Actions (Edit, Delete)
- Form fields: name
- Empty state: "No brands yet."

**OrderLive.Index:**
- Table columns: Order ID (truncated), Customer (name), Status (badge with color), Total, Items (count), Created At, Actions (View)
- No create button (orders created via workflow only)
- Filter: status dropdown (All, Pending, Confirmed, Fulfilled, Cancelled)
- Empty state: "No orders yet."

**OrderLive.Show (read-only + status transition):**
- Display: Order ID, Customer (name + email), Status (current badge), Total, Coupon (code or "—"), Created At
- Line items table: Variant (SKU + product name), Quantity, Unit Price, Subtotal
- Status transition: dropdown with only valid next states (based on current status), "Update Status" button
- No inline editing of line items or totals

All LiveViews:
- `mount/3`: load data using `Ash.read` with tenant context from connection
- `handle_event`: CRUD operations with `Ash.create`/`Ash.update`/`Ash.destroy`
- Flash messages on success/error
- Store-scoped via the `:store` pipeline's tenant context

## Key Ash Conventions

- All resources use `authorizers: [Ash.Policy.Authorizer]`
- All tenant-scoped resources use `multitenancy strategy(:context)`
- Actions use `primary?(true)` on explicit actions
- `require_atomic?(false)` on destroy actions with non-atomic changes
- `cascade_destroy` for parent→child deletion
- `after_action` for registry side effects
- Policies follow `authorize_if` chain pattern
- `Ash.read_one(..., authorize?: false)` for internal lookups (avoids circular auth)
- Decimal precision: `precision: 18, scale: 2` on all money fields
